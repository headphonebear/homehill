## What is this?

This directory contains the complete setup for automatic TLS certificate management using:

- **cert-manager**: Kubernetes certificate controller
- **Hetzner DNS Webhook**: DNS01 challenge solver for Let's Encrypt (Official Hetzner Cloud version)
- **Let's Encrypt**: Free, automated TLS certificates
- **Sealed Secrets**: GitOps-friendly encrypted API token storage

## Architecture

```
Certificate Request
    ↓
ClusterIssuer (letsencrypt-prod)
    ↓
DNS01 Challenge via Hetzner Webhook
    ↓
Webhook reads Hetzner Cloud API Token from Secret
    ↓
Webhook creates TXT record: _acme-challenge.homehill.de
    ↓
Let's Encrypt validates TXT record
    ↓
Certificate issued and stored in Secret
    ↓
Traefik uses Secret for TLS termination
```

## Files

- **Chart.yaml**: Helm chart definition with official Hetzner webhook dependency from `charts.hetzner.cloud`
- **Chart.lock**: Locked dependency versions (auto-generated by `helm dependency update`)
- **values.yaml**: Webhook configuration (groupName: `acme.homehill.de`)
- **templates/hetzner-rbac.yaml**: RBAC permissions for webhook to read secrets
- **templates/hetzner-token-sealed.yaml**: Encrypted Hetzner **Cloud Console** API token
- **templates/clusterissuer.yaml**: Let's Encrypt ClusterIssuer configuration
- **templates/wildcard-certificate.yaml**: Wildcard cert for `*.homehill.de`

## How it works

1. **Webhook Deployment**: ArgoCD deploys the official Hetzner webhook from Helm chart
2. **RBAC Setup**: Webhook gets permission to read secrets in cert-manager namespace
3. **Secret Unsealing**: Sealed-secrets controller decrypts the Hetzner Cloud API token
4. **ClusterIssuer Creation**: Cert-manager registers the Let's Encrypt issuer
5. **Certificate Request**: Wildcard certificate triggers DNS01 challenge
6. **DNS Challenge**: Webhook uses Hetzner Cloud API to create TXT record
7. **Validation**: Let's Encrypt checks TXT record and issues certificate (typically 30-60 seconds)
8. **Secret Storage**: Certificate stored in `homehill-wildcard-tls` secret
9. **Auto-Renewal**: Cert-manager renews 30 days before expiry

## 🔧 Configuration Details

### Official Hetzner Webhook

**Repository:** `https://charts.hetzner.cloud`  
**Chart Name:** `cert-manager-webhook-hetzner`  
**Version:** `0.6.3` (or latest stable)  
**API Endpoint:** `api.hetzner.cloud/v1` (Cloud Console API)

⚠️ **Important:** This is the **official Hetzner Cloud webhook**, not community forks! The old Robot DNS API (`dns.hetzner.com`) will be discontinued in **May 2026**.

### groupName Consistency

The `groupName` must be **identical** in:
- `values.yaml`: `webhook.groupName: acme.homehill.de`
- `clusterissuer.yaml`: `spec.acme.solvers[0].dns01.webhook.groupName: acme.homehill.de`

### Secret Configuration

**CRITICAL:** The official Hetzner webhook requires:
- **Secret name**: `hetzner`
- **Secret key**: `api-token` (not `token`!)
- **ClusterIssuer ref**: `tokenSecretKeyRef.name: hetzner`, `key: api-token`

**Example ClusterIssuer config:**
```yaml
config:
  tokenSecretKeyRef:
    name: hetzner
    key: api-token  # Must be "api-token" for official webhook!
```

### API Token Requirements

- Token must be from **Hetzner Cloud Console** (not old DNS Console)
- Go to: Hetzner Cloud Console → Your Project → Security → API Tokens
- Create token with **Read & Write** permissions
- Token needs access to DNS zones

## 🔄 Migration from vadimkim Webhook

If you're migrating from the old community webhook, these are the key changes:

### Chart Dependency

**OLD (vadimkim):**
```yaml
dependencies:
  - name: cert-manager-webhook-hetzner
    version: "1.3.3"
    repository: https://vadimkim.github.io/cert-manager-webhook-hetzner
```

**NEW (Official Hetzner):**
```yaml
dependencies:
  - name: cert-manager-webhook-hetzner
    version: "0.6.3"
    repository: https://charts.hetzner.cloud
```

### Secret Key Name

**OLD:** `token`  
**NEW:** `api-token`

You **must** reseal your secret with the new key name!

### API Endpoint

**OLD:** `dns.hetzner.com/api/v1` (Robot DNS API, deprecated)  
**NEW:** `api.hetzner.cloud/v1` (Cloud Console API, official)

### Migration Steps

1. Update `Chart.yaml` with new repository and version
2. Run `helm dependency update` to fetch new chart
3. Reseal secret with `api-token` key (see below)
4. Update `clusterissuer.yaml` to reference `api-token` key
5. Delete stuck resources: `kubectl delete order,challenge,certificate -n <namespace> <name>`
6. Commit and push - ArgoCD will sync
7. New certificate should be `Ready` in ~30-60 seconds

## 🔐 SealedSecret Management

### Re-encrypting the Secret

If you need to update the API token or migrate to a new cluster:

```bash
# 1. Set your Hetzner Cloud API token
export HETZNER_TOKEN="your-cloud-console-token-here"

# 2. Create secret (dry-run, not applied!)
kubectl create secret generic hetzner \
  --namespace=cert-manager \
  --from-literal=api-token="${HETZNER_TOKEN}" \
  --dry-run=client \
  -o yaml > /tmp/hetzner-secret-temp.yaml

# 3. Get sealing certificate from cluster
kubectl get secret -n kube-system \
  -l sealedsecrets.bitnami.com/sealed-secrets-key=active \
  -o jsonpath='{.items[0].data.tls\.crt}' | base64 -d > /tmp/sealing-cert.pem

# 4. Seal the secret
kubeseal --format yaml \
  --cert /tmp/sealing-cert.pem \
  < /tmp/hetzner-secret-temp.yaml \
  > templates/hetzner-token-sealed.yaml

# 5. Clean up (IMPORTANT - contains plaintext token!)
rm /tmp/hetzner-secret-temp.yaml /tmp/sealing-cert.pem
unset HETZNER_TOKEN
```

**The sealed secret should look like:**
```yaml
apiVersion: bitnami.com/v1alpha1
kind: SealedSecret
metadata:
  name: hetzner
  namespace: cert-manager
spec:
  encryptedData:
    api-token: AgAA7wqcddxtK6pUzgsQ...
```

Note the key is `api-token`, not `token`!

## 🔥 Troubleshooting: Stuck Certificates

If a certificate is stuck in `Ready=False` for hours/days:

### Symptoms:
- Certificate status: `Ready=False`
- cert-manager logs: "DNS record not yet propagated"
- Webhook logs: `404 Not Found` or API errors
- Order status: `pending` for extended time

### Root Causes:
1. **Wrong webhook version** (vadimkim for old API, need official for new API)
2. **Wrong secret key** (using `token` instead of `api-token`)
3. **Wrong API token** (using old Robot DNS token instead of Cloud Console token)
4. **Stuck resources** from previous failed attempts

### Solution:

```bash
# 1. Delete stuck resources
kubectl delete order,challenge -n <namespace> --all
kubectl delete certificate -n <namespace> <cert-name>

# 2. Verify correct webhook version and config (see Migration section above)

# 3. Certificate will be recreated automatically
# Should become Ready in 30-60 seconds

watch kubectl get certificate,order,challenge -n <namespace>
```
